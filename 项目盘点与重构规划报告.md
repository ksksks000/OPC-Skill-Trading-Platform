## 一、项目现状概览

### 1.1 Maven 模块结构

```
sky-take-out (parent, packaging=pom)
  ├── sky-common    —— 通用工具/基础设施模块
  ├── sky-pojo      —— 数据模型模块（DTO/Entity/VO）
  └── sky-server    —— 可执行应用模块（Controller/Service/Mapper）
```

**模块依赖关系**：
- `sky-server` 依赖 `sky-common` + `sky-pojo`
- `sky-common` 与 `sky-pojo` 互不依赖

### 2.4 值得继承的架构模式

| 模式 | 体现 | OPC 平台应用 |
|------|------|-------------|
| **统一响应模式** | Result\<T\> + PageResult | 所有 API 返回统一格式 |
| **ThreadLocal 上下文模式** | BaseContext | 服务层无侵入获取当前用户 |
| **自定义注解 + AOP 模式** | @AutoFill + AutoFillAspect | 可扩展为 @AiLog（AI 调用日志）等 |
| **异常层级模式** | BaseException + 子类 + GlobalExceptionHandler | OPC 业务异常体系 |
| **双 JWT 拦截器模式** | Admin/User 分离密钥 | 卖家/买家/管理员三端分离 |
| **Properties + Util + Configuration 三件套** | OSS 为例 | 支付、AI 等新模块参照此模式 |
| **WebSocket 会话管理** | SID 绑定 + 广播 | 实时通知、AI 对话流 |

## 五、现有项目结构评估与重构规划

### 5.1 当前架构优势

| 优势 | 说明 |
|------|------|
| **模块分层清晰** | common/pojo/server 三模块分离，职责明确 |
| **接口驱动设计** | Service 层接口与实现分离，便于后续替换为远程调用 |
| **双端 Controller 分离** | admin/user 端分离，与 OPC 的卖家/买家/管理三端天然对应 |
| **AOP 切面模式** | @AutoFill 模式可扩展为更多切面（AI 调用日志、操作审计等） |
| **异常体系完整** | BaseException 层级 + 全局处理，可直接扩展 |

### 5.2 当前架构不足

| 不足 | 影响 | 严重程度 |
|------|------|---------|
| **Spring Boot 2.7 已 EOL** | 无安全补丁，无法使用 Spring AI（需 Boot 3.x + Java 17） | 🔴 严重 |
| **Fastjson 1.x 有安全漏洞** | 反序列化攻击风险 | 🔴 严重 |
| **sky-pojo 依赖 knife4j** | 纯数据模块不应携带框架依赖，微服务拆分时造成传递依赖污染 | 🟡 中等 |
| **WebMvcConfiguration 继承 WebMvcConfigurationSupport** | 覆盖 Spring Boot 自动配置，应改用 WebMvcConfigurer 接口 | 🟡 中等 |
| **AutoFillAspect 有 BUG** | UPDATE 操作的审计字段不会被填充（第 67 行条件判断错误） | 🟡 中等 |
| **OrderMapper.xml countByMap 查询 setmeal 表** | 疑似复制粘贴错误，应查 orders 表 | 🟡 中等 |
| **application-dev.yml 硬编码密钥** | MySQL/Redis/OSS/微信密钥明文存储 | 🟡 中等 |
| **无生产环境配置** | 缺少 application-prod.yml | 🟢 轻微 |
| **Redis 依赖重复声明** | sky-server/pom.xml 中 spring-boot-starter-data-redis 出现两次 | 🟢 轻微 |

### 5.3 Spring Boot 3.x 升级迁移清单

| 变更项 | 详情 |
|--------|------|
| Java 版本 | 升级至 **Java 17+**（Spring Boot 3.x 最低要求） |
| javax -> jakarta | 所有 `javax.servlet`、`javax.validation`、`javax.xml.bind` 改为 `jakarta.*` |
| MyBatis | 升级至 `mybatis-spring-boot-starter` **3.0.x** |
| Druid | 升级至 `druid-spring-boot-starter` **1.2.16+** |
| Knife4j | 升级至 **Knife4j 4.x**（基于 SpringDoc / OpenAPI 3） |
| PageHelper | 升级至 **PageHelper 2.x** |
| JJWT | 升级至 **0.12.x**，API 有重大变更（JwtParserBuilder 替代 JwtParser） |
| Fastjson | 替换为 **Fastjson2** 或直接使用 Jackson |
| Jackson | 移除 sky-pojo 中手动指定的 2.9.2 版本，由 Spring Boot 3.x 管理 |
| Apache POI | 升级至 **5.x** |
| 循环依赖 | 移除 `allow-circular-references: true`，重构消除循环依赖 |

### 5.6 微服务演进路线（远期规划）

当前单体架构适合初期快速开发，未来若需微服务化，建议演进路径：

```
sky-take-out (parent)
  ├── sky-common           (共享工具/常量/异常)
  ├── sky-pojo             (共享 DTO/VO/Entity —— 移除 knife4j 依赖)
  ├── sky-gateway          (新增：Spring Cloud Gateway + JWT 鉴权)
  ├── sky-user-service     (User + 认证)
  ├── sky-skill-service    (技能上架/下架/搜索 —— 替代菜品/套餐)
  ├── sky-order-service    (订单/交易/状态机)
  ├── sky-admin-service    (Employee + 报表 + 工作台)
  ├── sky-ai-service       (AI 撮合/推荐/辅助交付)
  └── sky-payment-service  (支付/回调)
```

**微服务化前必须完成的准备**：
1. 引入 RabbitMQ 解耦跨域服务调用（先在单体内用 MQ 解耦）
2. 拆分数据库（按域分库：用户库、技能库、订单库）
3. 移除 sky-pojo 的框架依赖，使其成为纯数据模块
4. 将 BaseContext（ThreadLocal）替换为显式参数传递
5. WebSocket 改为 Redis Pub/Sub 或 RabbitMQ Fanout 广播
6. 定时任务引入分布式锁（Redis / ShedLock）

## 七、总结

### 可复用基建（22 个核心文件）

JWT 鉴权体系、全局异常处理、统一返回封装、AOP 自动填充、阿里云 OSS、Redis 配置、WebSocket、密码加密 —— 这些通用基建质量较高，可直接或稍作适配后复用到 OPC 平台。

### 需删除的外卖代码（58 个文件 + 7 张表）

菜品/套餐/分类/购物车/地址簿/店铺状态 —— 这套外卖核心业务代码与 OPC 无关，需彻底清理。

### 需改造的代码（34 个文件）

用户模块（去微信化）、订单模块（去配送化）、报表模块（去菜品化）—— 核心骨架可复用，但需移除外卖特有字段和微信依赖。

### 最大风险点

**Spring Boot 2.7 -> 3.x 升级**是整个改造中工作量最大、风险最高的环节，涉及 Java 版本升级、javax->jakarta 命名空间迁移、以及大量第三方库的兼容性升级。建议优先完成此步骤，再进行业务改造。
